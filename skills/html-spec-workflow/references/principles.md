# 底层原则（来自 Thariq Shihipar）

> 这五条不是格言，是 skill 在执行时真正需要遵守的"操作系统"。
> 当生成结果让你犹豫"要不要再加点东西"时，回来看这一页。

---

## 1. HTML 让"人"重新回到回路里

**为什么重要**：模型读 Markdown 没问题。但 Markdown 上千行计划会让**人**走神、跳读、放弃评审。
人不参与 → AI 单方面发挥 → 错的方向被强化 → 最终产出质量崩溃。

**实操含义**：每生成一个 HTML 产物，都问一句"**普通同事打开这个，会不会愿意往下滚？**"
如果答案是"不会"，hero 区不够吸引人，立刻重做。

---

## 2. 你正在变成一个"算力分配者"（Compute Allocator）

**为什么重要**：在 AI 时代，开发者/PM 的核心工作不再是"亲手写代码"，而是**决定把模型的算力花在哪里**。
对齐越准、规划越好 → 最后落地的代码越精准。

**实操含义**：
- 不要纠结"我有没有亲自写这个组件"。问自己"我是不是把算力分到了最有杠杆的地方"。
- 大量算力应该花在**沟通、规划、可视化**上，而不是"让模型多写一遍 CRUD"。

---

## 3. 复杂度必须自证其价值（Complexity has to earn its keep）

**为什么重要**：丰盈思维会让人忍不住"再加点交互"、"再多个动画"、"再扩个功能"。
但每一层复杂度都有维护成本——哪怕是一次性产物，也会**吃掉用户的注意力**。

**实操含义**：
- 加任何元素前问："**它真的让用户决策更快/更好了吗？**"
- 答案不是肯定的，就删掉。
- "看起来很酷"不是合理理由。

---

## 4. 约束 + 信任 = 最佳 Prompt 配比

**为什么重要**：
- **过度约束**（"必须包含这 17 项，每项必须 200 字以上……"）会让模型变成复读机，扼杀其判断力。
- **过度放飞**（"随便做做"）会让模型抓不到方向，产出离用户期待越来越远。

**实操含义**：在每个 prompt 里同时给到：
- 必要的结构性约束（输出格式、关键内容、范围边界）。
- 留白措辞：`whatever is needed to give me maximum context` / `design the ideal interface for this problem`。
- 一句**明示的信任**：`Hey Claude, I trust you here.` ——不是仪式，是给模型"你有发挥空间"的信号。

---

## 5. 99% / 1% 法则

**为什么重要**：Thariq 自述生成的 token 中，**只有约 1% 进入了生产代码**；其余 99% 都花在规划、定制界面、状态更新、设计系统这些"丰富、漂亮、有时一次性"的脚手架上。

这不是浪费——这是**杠杆**。

**实操含义**：
- 不要为"花在沟通和规划上的 token"感到内疚。
- 当你犹豫"要不要再花点算力做个更好的 plan.html"时——做。
- 真正的浪费不是"产出 99% 的脚手架"，而是"跳过脚手架直接写代码，结果方向错了重写一遍"。

---

## 衍生原则：丰盈思维（Abundance Mindset）

token 便宜了，**为自己造一个临时定制工具**也不亏。微应用、临时图表、一次性脚本——做就做了，扔就扔了。

唯一的纪律是**别让一次性产物污染长期维护体系**：放在 `./outputs/`、`./tmp/` 或 `_archive/` 里，不要混进 src。

---

## 衍生原则：即时文档（Just-in-Time Documentation）

文档不必是"长期维护的圣经"。一份**用一次就丢**的 HTML 计划，可能比一份"维护了三年但没人看"的 wiki 更有价值。

判断标准是**当下这次决策的质量**，不是"这份文档能用多久"。

---

## 当 skill 在执行中犹豫时的决策树

```
我现在在生成的内容，对用户的下一步决策有帮助吗？
├── 是 → 继续
└── 否 → 删掉

它够吸引用户打开/滚动/点击吗？
├── 是 → 继续
└── 否 → 重做 hero / 加一个图 / 减少文字

它让我陷入了"想做完美而停滞"吗？
├── 是 → 立刻先出最小可看版本，让用户在浏览器里点开看
└── 否 → 继续

它产生了不该长期维护的产物吗？
├── 是 → 放到 outputs/ 或 tmp/，明确标记"一次性"
└── 否 → 继续
```
